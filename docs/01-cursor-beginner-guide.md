# Cursor 新手手册

> 目标：30 分钟内理解 Cursor 基本工作流，并完成第一次 AI 辅助修改。读完本章你应该能独立描述一个"小功能修改"任务并让 Cursor 完成它。

## 1. Cursor 是什么

Cursor 是一个 AI 优先的代码编辑器，基于 VS Code 构建，但定位已经从"编辑器 + 补全"演进为 **agentic coding 工作台**：你不只是让 AI 帮你补一行代码，而是把整个任务（改文件、跑测试、看输出、修错误）交给 Agent 执行，你负责描述任务和验收结果。

与"传统 IDE + Copilot 插件"的核心差异：

- **AI 是一等公民**：Chat、Agent、Tab 补全深度集成在编辑器里，能直接读你的代码库上下文。
- **Agent 能执行多步任务**：不只生成代码，还能创建/修改多个文件、运行终端命令、根据报错自我修正。
- **上下文管理是显式的**：你可以精确控制 AI 能"看到"什么（文件、目录、选中片段、终端输出）。

## 2. 安装与账号

1. 从 [cursor.com](https://cursor.com) 下载对应平台安装包（macOS / Windows / Linux）。
2. 首次启动登录账号。免费档有用量限制，Pro 订阅解锁更高用量与更强模型；具体档位与价格以 [官方文档](https://docs.cursor.com) 为准。
3. 如果你从 VS Code 迁移，首次启动可一键导入扩展、主题与快捷键设置。
4. 建议首次进入设置检查两处：
   - **模型选择**：新手用默认推荐模型即可，不必纠结。
   - **隐私模式（Privacy Mode）**：处理公司代码前先确认团队的合规要求。

## 3. 界面速览

Cursor 的界面在 VS Code 布局基础上增加了 AI 工作区。你需要认识六个区域：

| 区域 | 位置（默认布局） | 用途 |
| --- | --- | --- |
| **Editor** | 中央 | 写代码的地方；支持 Tab 补全与选中后 Inline Edit |
| **Chat** | 侧边面板 | 与 AI 对话：提问、解释、小改动 |
| **Agent Window** | 独立窗口/面板 | 管理多个 Agent 会话，支持并行任务（见 docs/02） |
| **Changes** | 面板内 diff 视图 | 审查 AI 的每一处修改，逐个接受或拒绝 |
| **Terminal** | 底部 | 运行命令；Agent 也会在这里执行命令 |
| **Browser** | 面板/标签 | 部分版本内置浏览器预览，供 Agent 做页面验证 |

界面细节随版本迭代较快，控件位置以你安装版本的实际界面和 [官方文档](https://docs.cursor.com) 为准。核心心智不变：**写代码在 Editor，派任务在 Chat/Agent，验收改动在 Changes，验证行为在 Terminal/Browser**。

## 4. Context：让 AI 看到正确的东西

AI 的输出质量首先取决于它看到了什么。Cursor 提供显式的上下文控制，常用五种：

- **文件引用**：输入 `@` 引用具体文件，如 `@src/utils/date.ts`。这是最常用、性价比最高的上下文。
- **目录引用**：`@src/components/` 让 AI 了解一个模块的整体结构。
- **选中片段（selection）**：在 Editor 里选中代码后发起 Chat/Inline Edit，AI 会以选中内容为焦点。
- **截图/图片**：把 UI 截图拖进对话，适合"照着这个样式改"类任务。
- **终端输出**：把报错信息粘贴（或引用终端内容）给 AI，比口头转述"报了个错"有效十倍。

新手法则：**你希望 AI 改哪个文件，就先 `@` 哪个文件**。不给上下文让 AI 自己猜，是产出跑偏的第一大原因。

## 5. Prompt 基础：四要素

一个可靠的任务描述包含四个要素。缺一个，AI 就要猜一个。

| 要素 | 回答的问题 | 示例 |
| --- | --- | --- |
| **目标** | 做什么、做到什么程度 | "给 `formatDate` 增加相对时间输出（'3 天前'）" |
| **上下文** | 需要看哪些文件/背景 | "@src/utils/date.ts @src/utils/date.test.ts" |
| **约束** | 不能动什么、遵守什么 | "不改现有函数签名；不引入新依赖" |
| **验收** | 怎么算完成 | "npm test -- date 全部通过；补充 3 个新用例" |

对照示例——同一个需求的两种写法：

```text
❌ 帮我优化一下日期处理

✅ 目标：给 @src/utils/date.ts 的 formatDate 增加相对时间模式（今天/昨天/N 天前）。
   约束：不改现有调用方；不加新依赖。
   验收：npm test -- date 通过，为新模式补 3 个测试用例。
```

## 6. 六种常见使用模式

### 6.1 解释代码

选中代码片段或 `@` 文件后问"这段代码做什么、有什么坑"。适合接手陌生项目。追问比一次问全更有效。

### 6.2 补全与小改动

Tab 补全处理"你知道要写什么"的场景；选中代码用 Inline Edit（快捷键以当前版本为准）处理"这一小段要改"的场景。改动超过一个文件时升级到 Chat/Agent。

### 6.3 修 bug

先给现象和复现路径，**要求 AI 先分析根因、你确认后再改**：

```text
现象：提交表单后页面白屏。控制台报错：<粘贴完整报错>。
先定位根因并解释，不要直接改代码。
```

"先 trace 后修改"能避免 AI 把症状当病因、越修越乱。

### 6.4 写测试

给出被测文件和现有测试风格，让 AI 照着写：

```text
为 @src/utils/date.ts 的 formatDate 写单元测试，
风格参考 @src/utils/string.test.ts，覆盖边界：无效日期、跨年、时区。
```

### 6.5 改 UI

截图 + 具体描述比纯文字有效：

```text
（附当前页面截图）把这个卡片的间距改成 16px，标题字号调大一级，
移动端（<768px）改为单列。只改 @src/components/Card.tsx 及其样式文件。
```

### 6.6 让 AI review 你的代码

提交前把 diff 或文件交给 AI 审查："检查 @src/api/order.ts 的错误处理是否有遗漏，列出问题，不要直接改"。注意这只是自查辅助，不能替代正式的 reviewer gate（见 docs/04）。

## 7. 新手四大错误

### 错误一：任务太大

一次让 AI"重构整个模块 + 加新功能 + 顺便修 bug"，产出必然失控。

```text
❌ 重构 src/api 目录，顺便把报错处理统一一下，再加上重试机制

✅ 第一步：只给 @src/api/client.ts 增加统一的错误类型 ApiError，
   不改其他文件。验收：现有测试全过。
   （完成验收后再派下一步）
```

### 错误二：上下文缺失

不给文件引用，让 AI 凭空猜项目结构。

```text
❌ 把用户头像组件改成圆形的

✅ @src/components/Avatar.tsx @src/styles/avatar.css
   把头像改为圆形（border-radius: 50%），保留现有尺寸变体 sm/md/lg。
```

### 错误三：没有验收标准

不说怎么算完成，AI 说"已完成"你也无从判断。

```text
❌ 修一下登录页的 bug

✅ 修复：连续点击登录按钮会重复提交。
   验收：快速点击 5 次只发出 1 个请求（Network 面板验证）；
   npm test -- login 通过。
```

### 错误四：让 Agent 猜环境

不告诉 AI 项目怎么跑、用什么包管理器、测试命令是什么。

```text
❌ 改完帮我测一下

✅ 项目用 pnpm；启动：pnpm dev（端口 5173）；测试：pnpm test。
   改完运行 pnpm test 并贴出结果。
```

## 8. 实操：完成你的第一次 AI 辅助修改

把前面的要素串起来，完整走一遍：

1. **挑任务**：选一个 30 分钟内能人工验证的小改动（改文案、加一个工具函数、修一个已定位的 bug）。
2. **写任务**：按"目标/上下文/约束/验收"四要素写进 Chat，`@` 上相关文件。
3. **看 diff**：AI 完成后，在 Changes 面板逐文件审查。**看不懂的改动就问**："第 42 行为什么要加这个判断？"
4. **接受或拒绝**：只接受你理解且需要的改动；跑偏的部分拒绝并补充说明重派。
5. **验证**：亲自运行验收命令（测试/启动/点页面），不要只信 AI 的"已完成"。
6. **复盘**：如果结果不对，回看你的任务描述缺了四要素中的哪一个。
