# 成本与风险控制

> 目标：读完本章你能在派发前预估任务的成本档位、判断哪些任务根本不值得派发，并用清单规避多 Agent 协作的常见翻车点。

## 1. 多 Agent 的成本模型

成本有三类，只有第一类是显性的：

- **token 成本**：模型调用消耗。与任务描述质量强相关——模糊任务导致 agent 多轮试错，烧的都是 token。
- **时间成本**：交接（写包、派发、读回传）、等待、review 的人力时间。
- **返工成本**：范围跑偏、验收不明、并行冲突导致的重做。三类里最贵，且往往被记在"模型不行"头上。

**引擎选择即计费选择**：`engine: codex-cli` 跑在本机 `codex exec`，模型与用量归 OpenAI 订阅；`engine: cursor-subagent` 跑在 Cursor 后台 subagent，模型与用量归 Cursor 订阅。把二者混用却不留档，会让成本归因不可审计：你以为在花 Codex 额度，实际可能在消耗 Cursor 用量，或反过来。因此 task package / review packet / 派发话术都应记录实际 engine。

并行放大所有三类：N 个 agent 是 N 份 token；每个 agent 都要一份完整上下文；产出要 N 次 review。**并行的收益只有时间，成本却是全方位的**——这是"能串行不并行"的经济学依据。

## 2. 按路径的成本分级

对应 routing matrix（docs/03 第 4 节）的成本列：

| 档位 | 典型路径 | 主要开销 | 省钱手段 |
| --- | --- | --- | --- |
| 低 | Cursor 直接问答 / 小改动 | 少量对话轮次 | 问题问具体，附上 `@` 文件 |
| 中 | Codex 单任务、Plan 拆解、Multitask 只读调研 | 交接成本 + agent 多轮执行 | 任务包四要素齐全，一次说清；调研限定输出格式 |
| 高 | 并行 worktree 开发、跨端联调、上线前检查 | N 份执行 + N 次 review + 环境成本（每个 worktree 独立依赖/构建） | 压缩并行度；合并可串行的子任务；gate 抽查代替全量复核 |

### 什么任务不值得派发

判断法一句话：**沟通成本 > 执行成本的任务，自己做**。具体信号：

- 描述任务包所需时间超过自己改所需时间（改一行文案、重命名一个变量）。
- 验收标准写不出来（说明需求还没想清楚，派出去必返工——先澄清，不是先派发）。
- 强依赖只有你脑子里才有的隐性上下文，且外化成本高。
- 一次性的探索式操作（"我就想看看点开是什么样"）。

派发的规模效应在于：任务包写一次，执行、审查、留档、复盘都复用它。复用不到两次的沟通投入，通常不划算。

## 3. 并行的隐性成本

- **冲突返工**：两个任务改同一文件，后合并者解决冲突，解决错了还要返工。协议对策：并行写任务强制 `isolation: worktree`（机制消灭同仓冲突，代价转为可预估的 worktree 环境成本）。
- **重复上下文**：每个并行 agent 都要喂一遍项目背景。对策：公共上下文写进 task package 的"上下文"节引用文件路径，不在对话里重复粘贴。
- **review 放大**：N 个并行产出在短时间内同时到达 gate，review 质量随疲劳下降。对策：控制同时在途（in-flight）的任务数——gate 消化能力才是并行度的真实上限，不是 agent 数量。

## 4. 风险控制清单

立项书第 12 节风险表逐条展开为操作建议：

| 风险 | 操作建议 |
| --- | --- |
| Cursor 功能更新快，手册过期 | 引用功能行为时附官方链接；发现表述与实际不符，按 `docs/changelog-watch.md` 机制登记核验；阅读时留意每章 `last_verified` 日期。 |
| 多 Agent 改同一文件冲突 | 派发前检查在途任务清单，`scope.allow` 有交集且时间并行 → 强制各自 `isolation: worktree`；无法隔离就串行。 |
| 成本不可控 | 填 task package 时对照成本档位（第 2 节）；高档位任务先问"能不能降档"（缩范围、去并行）；单任务连续两轮 FAIL 即止损（docs/04 状态机 Abandoned）。 |
| 上下文泄漏 | 见第 5 节上下文安全。 |
| reviewer / executor 输出不一致 | 谁也不信，以验收标准和本地验证为准（human gate 亲自跑命令，见 docs/04 第 5 节）。 |
| 新手误把 Agent 当自动驾驶 | 每个任务保留 human gate 触点；不可逆操作（删数据、发布、force push）永远不进 agent 的自主权限。 |

### 敏感边界与强制 gate 规则

涉及以下五类的任务，协议强制要求（不是建议）：

**auth（认证授权）/ payment（支付）/ data（数据迁移与删除）/ deployment（部署发布）/ shared contract（跨端共享契约）**

- `risk_level` 至少 `medium`，通常 `high`。
- `gate_required: human`，human gate 不可省略（双工具与三工具配置下相同）。
- `isolation` 至少 `branch`，并行时必须 `worktree`。
- `scope.deny` 显式列出不许触碰的敏感路径（如 `src/auth/**`、`.env*`、migration 目录），不要依赖 executor 的自觉。

## 5. 上下文安全

task package 与 review packet 会被复制、留档、可能开源——按"这份文件将来会公开"的标准写。

| 禁止出现 | 替代写法 |
| --- | --- |
| API key、token、密码明文 | 写环境变量名：`使用 $STRIPE_API_KEY（本地 .env 配置）` |
| 内部服务 URL、生产库连接串 | 写占位符与获取方式：`staging 地址见团队 1Password 条目 xxx` |
| 真实用户数据、PII | 用构造的假数据示例说明结构 |
| 完整 `.env` 内容作为"上下文" | 只给字段名清单与用途说明 |

review packet 贴命令输出前扫一眼：报错堆栈和日志里经常裹着连接串与 token，贴之前脱敏。

## 6. Rollback 思维

多 Agent 工作流的最后一道保险不是"不出错"，而是**任何时候都有便宜的退路**：

- **branch / worktree 的可丢弃性**：主工作区从未被修改，`git worktree remove --force` + 删分支就是完整回滚，成本近乎为零。这就是高风险任务强制隔离的根本原因——买的就是这条退路。
- **merge 前的最后检查**（不可逆操作前的 30 秒）：`git diff main...task/<id> --stat` 扫文件清单有无越界；确认验收抽查真的跑过；确认这是 `worktree.merge_by` 授权的合并。
- **merge 之后**：worktree 删了但分支历史还在，`git revert` 仍可退。真正不可逆的是部署、数据变更、对外发布——这些永远留在 human gate 手里。
